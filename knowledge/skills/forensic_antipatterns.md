# 取证做题 反模式集

> **作用**：把"复盘里发现的错"抽象为**通用反模式**，每个反模式有"症状 / 根因 / 解药"。
> **维护**：每次复盘新增 1-3 条。
> **第一来源**：2026FIC computer_analyst v1 复盘的 6 大错因。

---

## AP-001 协议识别滞后

**症状**：拿到加密文件就直接做密码学逆向 / 暴破。

**根因**：跳过了"取证软件 magic 检测"这一步，把工具误认为加密器。

**典型例子**：
- 2026FIC vc 容器：火眼直接弹"需 VeraCrypt 解密"提示，被忽略；反而花 8 小时反汇编 SampleVC.exe
- BitLocker 容器被当成"自定义 AES 卷"
- LUKS 头被当成"加密 ELF"

**解药**：
1. 任何加密文件 **首先** `file` + `binwalk -B` + `ent` 三步识别
2. 火眼/Autopsy 提示一律按提示走，**不要先入为主**
3. 工具命名本身就是提示（`Sample` 前缀 = 示例 / 加载器，不是逆向核心）

---

## AP-002 语义实体绑定污染

**症状**：把题面 X 直接绑死到已经看见的资产 Y 上，再围绕 Y 深挖。

**根因**：T0 关键词搜索没做就开始干活，导致整个搜索域被一开始的猜测污染。

**典型例子**：
- 2026FIC："保险柜密码" 被绑到 "vc 容器密码"，从未搜过 "保险" 关键词
- 实际 `/root/文档/zhongyao/保险箱的秘密.et` 完全独立存在，但因为绑定污染从没被发现

**解药**：
1. **每道题独立做 T0**：题面所有名词 → ripgrep 全检材
2. **不要复用上一题的实体猜测**
3. 题面术语用**直译关键词**，不要语义二次解读
4. 每个实体先 `path-level` 命中，再 `content-level` 验证；两步独立

---

## AP-003 跨角色协作滥用

**症状**：自己检材没穷尽就 "转给其他角色" / "标记为跨检材"。

**根因**：把"协作"当成"借口"逃避深挖；把"binary 角色"当成万能投递箱。

**典型例子**：
- 2026FIC Q7 apiKey 找不到就转 server，实际答案就在 PC 的 `~/.config/uos-ai/data.db`
- Q9/Q10 找不到就转 stego_crypto，实际答案分别在 vc 解密后的 mp4 和 PC 的 .et 文件

**解药**：
1. **协作只在两种情况启用**：
   - (a) 题面明确指向其他检材（"手机里的 X" "服务器的 Y"）
   - (b) 自己穷尽了所有合理路径后仍 0 命中
2. "在我的检材里没找到" ≠ "答案不在我的检材里"
3. 转手前必须附"已尝试的搜索域清单"，不空手转
4. 每次转手要回答："还有哪 3 个目录我没扫？"

---

## AP-004 必扫目录遗漏（OS 自带应用）

**症状**：搜遍浏览器/邮件/通用配置，独漏系统自带应用。

**根因**：不熟悉特定 OS 的预装应用清单。

**典型例子**：
- Linux deepin: UOS AI / 语音记事本 / 文件保险柜
- Win11: Copilot / Edge AI / Win+H 语音
- macOS 15+: Apple Intelligence
- Android: Google AICore / Gemini

**解药**：
1. 维护 `must_scan_dirs.md`（按 OS 分章），新 OS 加章
2. 拿到镜像第一步 `dpkg -l | grep -i ai` / `ls /usr/share/applications/` / 查 dock 配置
3. 把"系统自带 AI 助手"列为单独优先级（§11）

## AP-005 弱密钥扫描跳过

**症状**：拿到 RSA 公钥 (n, e)，搜了一圈"私钥文件"找不到就放弃。

**根因**：以为"私钥不在文件系统 = 私钥不存在"，没意识到 CTF 设计就是要算出来。

**典型例子**：
- 2026FIC Q4 public.txt 的 (n, e=65537) — p、q 接近 → Fermat 10 行代码 1 秒出私钥
- 多次见到 e 异常大 → Wiener 攻击未尝试

**解药**：
1. 公钥到手 **强制走 `weak_crypto_kit.md` §1 全表**（12 个攻击点）
2. 永远不要因"私钥不在文件系统"而放弃
3. **factordb.com** 是 30 秒成本的兜底
4. RsaCtfTool 一行命令覆盖大部分场景

---

## AP-006 跨检材密码复用未尝试

**症状**：在检材 A 加密容器上单独暴破，没把检材 B 已发现的密码倒过来试。

**根因**：把每个检材的加密视为独立密码学问题。

**典型例子**：
- 2026FIC vc 容器密码 `9ed2@99y8.com.cn` 直接写在手机便签里 — 但 PC 角色花了 16 小时暴破 16-char ASCII 候选
- 服务器 root 密码常等于 PC 用户登录密码

**解药**：
1. 任何检材发现密码 → **立刻 broadcast** 到所有跨检材任务（hub finding 或 git inbox）
2. 加密容器**先用所有已知密码列表试一遍**，再考虑暴破
3. **30% 复用法则**：同一用户的密码 30%+ 概率复用，必试
4. 候选密码也包括变体（大小写、+ 数字尾、+ 年份）

## AP-007 自定义加密深挖（不该深挖）

**症状**：看到二进制里有"加密"就反编译追算法。

**根因**：把 sample loader / GUI wrapper 误判为密码学核心。

**典型例子**：
- SampleVC.exe 只是个 `VirtDisk.dll` 调用器，被当成核心加密器深挖
- 大量 "Sample" / "Demo" / "Loader" 命名的程序都是包装层

**解药**：
1. 反编译前先看**文件名语义**（Sample / Demo / Loader / Wrapper / Helper 都是包装）
2. 看 imports 表 — 大量调用系统 API 的就是包装，不是核心
3. 程序大小 < 200KB 的几乎不可能含完整密码学算法
4. **真正的密码学核心**：往往在 dll / so / 内嵌 lib，不在主 exe

---

## AP-008 隐写宿主不识别

**症状**：看到 mp4 / docx / png 默认它是普通文件，不查隐写。

**根因**：缺少"隐写宿主全景"知识；把扩展名当类型。

**典型例子**：
- 2026FIC Q9 mp4 `stco` 表加 1337 — 不知道 mp4 box 可作隐写载体
- Q10 WPS .et `Shape.AlternativeText` 数字串 — 不知道 OLE 可这样用
- PNG tEXt chunk 长字符串 — 当成普通元数据

**解药**：
1. 维护 `stego_host_atlas.md`（10+ 类宿主，§1-§9）
2. 题面有"隐藏""加密""不可见""异常"关键词 → 强制走 §10 通用解码动作
3. **任何容器格式都默认可能是隐写宿主**，不只是图片

---

## AP-009 时间盒失控

**症状**：单题花 8+ 小时还在做。

**根因**：没有 30 分钟硬上限；陷入"再尝试一种方法"循环。

**典型例子**：
- vc 容器暴破 16 小时（密码本不在 PC）
- RSA 公钥分析 5 小时（应该 1 秒 Fermat）

**解药**：
1. **单题硬上限 30 分钟**；超时立即升级或跳过
2. 三遍扫题：5 min × 10 → 15 min × 剩下 → 无上限 × 最难
3. 第二遍开始才允许反汇编 / 弱密钥 / 隐写
4. 升级标准：30 min 仍 0 进展 = 立即转协作 / 跳过

---

## AP-010 角色越界（用错角色思维）

**症状**：用 binary 角色的反汇编思维做 computer 角色的痕迹取证题。

**根因**：分工太干净，把"看到加密文件"和"反汇编"无条件绑定。

**典型例子**：
- 2026FIC computer 角色把 vc / get_token_linux / .et 全推给 binary
- 实际 computer 角色就该会 mp4 atom / OLE / 轻量 ELF strings

**解药**：
1. **取证 = 找用户的痕迹**，不是密码学谜题
2. 先做"找痕迹"，找不到再做"破密码"
3. 每个角色都要有最小工程能力：
   - computer: OLE / sqlite / mp4 atom / strings / hex
   - mobile: leveldb / sqlcipher / apk dex
   - server: docker / nginx / mysql
   - binary: ghidra / ida / unicorn
4. 协作不是替代基础工程能力

---

## AP-011 火眼/取证软件 UI 不优先

**症状**：直接命令行挖深路径，不开取证软件 GUI。

**根因**：以为"GUI 是给新手用的"，命令行更专业。

**典型例子**：
- 2026FIC Q3 黄金商家联系方式：火眼仿真 → 桌面应用 → 语音记事本 → 一眼看见
- 用 sqlite 解析 `deepin-voice-note1.0.db` 也能解，但花 10x 时间

**解药**：
1. **T1 强制开取证软件 UI**（火眼 / Autopsy / FTK Imager / X-Ways）
2. 先看"系统信息""安装应用""桌面""最近文件""便签"几个 panel
3. UI 解决不了再上命令行
4. UI 比命令行**对常见取证场景更优**，因为内置了 hundreds of parsers

---

## AP-012 把 hash 校验当作暴破入口

**症状**：看到 SHA1 / MD5 hash 就开始暴破。

**根因**：以为 hash 是密码本身的 hash；不区分"hash 用途"。

**典型例子**：
- 2026FIC vc 容器周边有 MD5 + SHA1 两个 hash
- 误以为两个 hash 都是密码 hash（"双 hash 约束"）
- 实际可能一个是文件完整性校验，另一个才是密码 hash

**解药**：
1. 拿到 hash 先**追溯它的用途**（来自哪个文件、哪段代码）
2. **hash 仅供校验**，密码必须从用户记忆痕迹获得（便签/邮件/聊天）
3. 在线 rainbow 库（hashes.com / nitrxgen）30 秒兜底
4. 单 hash 暴破 ≠ 多 hash 联立约束 — 多约束应先短 hash 过滤

## 反模式应用流程

每次开始解题前快速过：

```
□ AP-009 设 30 min 计时器了吗？
□ AP-002 T0 关键词原文搜索做了吗？
□ AP-011 取证软件 UI 开了吗？
□ AP-001 加密文件先做协议识别了吗？
□ AP-004 OS 自带应用必扫目录扫了吗？
□ AP-005 公钥到手跑了 weak_crypto_kit 吗？
□ AP-006 已知密码列表试过所有加密容器了吗？
□ AP-008 容器格式查了 stego_host_atlas 吗？
□ AP-003 我自己穷尽了所有路径再决定转手？
```

打勾 < 8/9 不开始重活。
