# worklog/ — 工作日志

> 每次 Cascade 会话结束（或暂停）时，留一份日志到此目录。下次会话**第一件事**读最新的一份。

---

## 命名规则

```
worklog/YYYY-MM-DD_NN_<主题>.md
```

- `YYYY-MM-DD`：日期
- `NN`：当天第几次（01, 02, ...）
- `<主题>`：一句话概括（snake_case）

示例：
- `2026-04-24_01_三模式架构与分发包.md`
- `2026-04-25_01_FIC实战.md`
- `2026-04-25_02_FIC赛后总结.md`

按文件名字典序排就是时间顺序，最新的在最下面。

---

## 每份日志的骨架（强制字段）

```markdown
# <主题>

**时间**：YYYY-MM-DD HH:MM（CST/UTC+8）
**时长**：大约 N 小时
**模式**：SOLO / COOP / TEAM / 项目维护
**本次目标**：一句话

## 做了什么
- …（bullet，动词开头）

## 产出的文件
- `path/to/file` — 做什么用的

## 改动的文件
- `path/to/file` — 改了哪个函数/段落

## 遇到的问题 & 解决
- **问题**：…
  **解决**：…

## 悬而未决
- …（明确列出，下次接手不要重复踩）

## 下次建议从这里开始
- 具体动作：…
- 待验证：…
```

---

## Cascade 开场读哪份？

```
ls -t worklog/*.md | head -1
```
读取最新的那份 → 上下文对齐后再干活。

若是跨多天，读最近 2-3 份，避免遗漏"前天的悬而未决"。

---

## 写入时机（强制）

| 时机 | 强制 |
|------|------|
| 用户说"暂停/休息/晚安" | ✅ 必写 |
| 连续工作 > 2 小时 | ✅ 必写 |
| 架构/规则发生改动 | ✅ 必写 |
| 单题答案（那是 `cases/<比赛>/answers.md` 的事） | ❌ 不写 |
| 每 5 题批次（那是 `wp_batches/` 的事） | ❌ 不写 |

**写日志 ≠ 写答案**。本目录只记录 **"项目/会话级"** 事件，不记录单题。
