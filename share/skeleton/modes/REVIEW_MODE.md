# 复盘模式 (REVIEW MODE)

> 比赛已结束，进入复盘和沉淀阶段。**不赶时间，重在总结规律**。

---

## 模式特征

| 维度 | REVIEW 模式 |
|------|------------|
| 时间压力 | 无 |
| 主要目标 | 总结教训、沉淀套路、完善知识库 |
| 问用户频率 | 中（确认复盘结论是否准确） |
| 主动决策 | 中（可以自主整理，但重要结论要确认） |

## 进入条件

用户明确说"比赛结束"或"进入复盘模式"。

## 复盘 SOP

### 1. 生成成绩总结
- 按板块统计：已解/未解/得分/失分
- 输出 `wp_batches/INDEX.md`（答案总索引表）

### 2. 生成复盘报告
- 写 `RETROSPECTIVE.md`：成功路径 + 失误分析 + 时间分配
- 写 `LESSONS_LEARNED.md`：可迁移的通用教训

### 3. 沉淀解题套路
- 每种新解出的题型 → `AI_BRAIN/solved_patterns/<题型>.md`
- 每个新发现的工具用法 → `AI_BRAIN/tool_inventory.md`
- 每个踩过的坑 → `worklog/AI_DIAGNOSTICS.md`

### 4. 更新工具清单
- 标注哪些工具实际用了、效果如何
- 标注哪些工具该装没装、导致失分
- 标注哪些工具装了没用

### 5. 准备分享版（可选）
如果用户要求"做成分享版"：
- 建 `share/` 子目录
- 含：骨架 + 知识库 + 安装脚本 + 启动手册 + 参考案例

## 质量标准

- 每条教训必须有**具体案例**支撑（"第 X 题因为 Y 导致 Z"）
- 套路必须有**可复现命令**
- 工具评价必须基于**实际使用经验**，不能凭空推测

## 退出条件

- 用户说"开始新比赛"→ 切回 SOLO/COOP/TEAM
- 所有复盘文件写完 + 用户确认
